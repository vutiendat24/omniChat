# TASK 1.3.1.1 - Kafka Topics Design & Configuration

## 1. Event Storming Result

Phân tích các event dựa trên Bounded Context của OmniChat:

*   **Conversation Domain:** `conversation.created`, `conversation.assigned`, `conversation.closed`, `conversation.transferred`
*   **Customer Domain:** `customer.created`, `customer.updated`
*   **Integration Domain:** `integration.message`, `integration.status`
*   **WebSocket Domain:** `websocket.message.push`
*   **Notification Domain:** `notification.send`

## 2. Topic Design Table

Naming convention: `omnichat.<domain>.events`

| Topic Name | Producer | Consumer | Partitions | RF (Dev/Prod) | Retention | DLQ Topic |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `omnichat.conversation.events` | Conversation | Routing, WS | 6 | 1 / 3 | 7 days | Yes |
| `omnichat.customer.events` | Customer | Conversation | 6 | 1 / 3 | Compact | Yes |
| `omnichat.integration.events` | Integration | Conversation | 6 | 1 / 3 | 3 days | Yes |
| `omnichat.websocket.events` | Routing, Conv | WebSocket | 6 | 1 / 3 | 1 day | Yes |
| `omnichat.notification.events`| Routing | WS, Email | 3 | 1 / 3 | 1 day | Yes |

*Ghi chú: Partitions được thiết lập là 6 cho các topic chính để cho phép scaling tối đa 6 instances của consumer (đáp ứng >100k messages/hour thoải mái).*

## 3. Partition Strategy

*   Sử dụng **Default Partitioner** (Hash-based partitioner) của Kafka.
*   Partitioner sẽ hash Message Key để tính toán ra partition: `hash(key) % num_partitions`.
*   Tất cả các message có cùng một key sẽ luôn được đẩy vào cùng một partition, đảm bảo **Strict Ordering**.

## 4. Key Design Strategy

Để đáp ứng ràng buộc "Message ordering phải được đảm bảo theo ConversationId", chiến lược đặt Key như sau:

*   **Topics:** `omnichat.conversation.events`, `omnichat.integration.events`, `omnichat.websocket.events`
    *   **Message Key:** `ConversationId` (String/UUID)
    *   *Mục đích:* Bất kỳ event nào sinh ra trong quá trình trao đổi của một cuộc hội thoại đều được đưa vào cùng 1 partition. Khi Consumer đọc từ partition đó, các event sẽ tuần tự theo đúng trình tự thời gian sinh ra.
*   **Topic:** `omnichat.customer.events`
    *   **Message Key:** `CustomerId` (String/UUID)

## 5. Retention Policy

*   **Time-based Retention:** Cho các topic về luồng tin nhắn và notification, giới hạn thời gian (1, 3, hoặc 7 ngày) vì dữ liệu này sau khi xử lý thường được lưu trữ vào Database (MySQL/MongoDB), không cần giữ mãi trên Kafka để tiết kiệm disk space.
*   **Log Compaction:** Dành riêng cho `omnichat.customer.events`. Bật `cleanup.policy=compact` để Kafka chỉ giữ lại version mới nhất của mỗi `CustomerId`. Điều này giúp các service khởi động sau có thể nạp lại state của user một cách nhanh chóng.

## 6. DLQ Design (Dead Letter Queue)

Sử dụng pattern **Non-blocking Retry & DLQ**:
*   Khi Consumer xử lý lỗi (Exception như DB timeout, NPE), message sẽ được retry nội bộ.
*   Sử dụng Spring Kafka `DefaultErrorHandler` kết hợp `DeadLetterPublishingRecoverer`.
*   Sau 3 lần retry thất bại, message được tự động đẩy vào DLQ.
*   **DLQ Naming Convention:** `<original-topic-name>.<consumer-group-id>.DLT` (Ví dụ: `omnichat.integration.events.conversation-service.DLT`)
*   Bao gồm header chứa Exception Stacktrace để phục vụ tracing & re-processing.

## 7. Kafka Config (Broker level)

Khuyến nghị cấu hình cho Production:
```properties
# Replication
default.replication.factor=3
min.insync.replicas=2

# Retention
log.retention.hours=168      # 7 days

# Message Size
message.max.bytes=5242880    # 5MB (đáp ứng rich media urls hoặc text dài)

# Compaction
log.cleaner.enable=true
```

## 8. Spring Boot Producer Config

```yaml
spring:
  kafka:
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
      acks: all              # Ensure no data loss (Requires min.insync.replicas)
      retries: 3
      properties:
        enable.idempotence: true                       # Prevent duplicate messages
        max.in.flight.requests.per.connection: 5       # Keep ordering with idempotence
```

## 9. Spring Boot Consumer Config

```yaml
spring:
  kafka:
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.ErrorHandlingDeserializer
      properties:
        spring.deserializer.value.delegate.class: org.springframework.kafka.support.serializer.JsonDeserializer
        spring.json.trusted.packages: "com.omnichat.*"
      group-id: ${spring.application.name}-group
      auto-offset-reset: earliest
      enable-auto-commit: false    # Use manual ack for exactly-once processing semantic
    listener:
      ack-mode: MANUAL_IMMEDIATE
      concurrency: 3               # Scale up reading threads (max = partition count)
```

## 10. Topic Creation Script

Dùng script `create-topics.sh` để khởi tạo trên môi trường Local (Docker):

```bash
#!/bin/bash
# Wait for Kafka to be ready
sleep 10

# Default configuration for Local (RF=1)
RF=1
PARTITIONS=6

TOPICS=(
  "omnichat.conversation.events:$PARTITIONS:$RF"
  "omnichat.customer.events:$PARTITIONS:$RF"
  "omnichat.integration.events:$PARTITIONS:$RF"
  "omnichat.websocket.events:$PARTITIONS:$RF"
  "omnichat.notification.events:3:$RF"
)

for topic_config in "${TOPICS[@]}"; do
  TOPIC_NAME=$(echo $topic_config | cut -d: -f1)
  PARTS=$(echo $topic_config | cut -d: -f2)
  REP_FACTOR=$(echo $topic_config | cut -d: -f3)
  
  echo "Creating topic: $TOPIC_NAME"
  docker exec -it kafka kafka-topics.sh --create --if-not-exists \
    --bootstrap-server localhost:9092 \
    --topic $TOPIC_NAME \
    --partitions $PARTS \
    --replication-factor $REP_FACTOR
done

# Config log compaction for customer topic
docker exec -it kafka kafka-configs.sh --alter \
  --bootstrap-server localhost:9092 \
  --entity-type topics \
  --entity-name omnichat.customer.events \
  --add-config cleanup.policy=compact
  
echo "All topics created successfully!"
```
